Milestone 2

Panic, Test Harness, and QEMU Exit Relationships

This diagram shows how the current implementation separates the normal kernel path from the test-kernel path. The most important design rule is that qemu::exit_* exists only for the test harness world, not for ordinary kernel logic.

Normal Kernel Path

What happens when the kernel boots without the test-kernel feature

Entry

kernel_main()

Clears BSS, initializes the console, prints the boot banner, then waits forever.

Output

console

Provides print! and println!, then delegates actual byte output to the UART driver.

Failure Policy

panic::handle()

Prints the panic message and stops in a waiting loop. It does not try to exit QEMU.

main.rs → console::print → driver::uart → serial output

Test-Kernel Path

What changes when the kernel is built with --features test-kernel

Feature Gate

main.rs

Instead of printing the normal boot banner, the boot path immediately transfers control to the test runner.

Harness

test::run()

Prints running N tests, executes each test case, and marks successful tests as ok.

Host Signal

driver::qemu

Reports success or failure back to the host by writing to the QEMU test finisher device.

main.rs → test::run() → qemu::exit_success() → QEMU exits with pass status

Panic Behavior Split

Normal Mode Panic

Normal kernel behavior should look like a real kernel failure path: print the panic message, then stop. No emulator-specific exit shortcut is used here.

panic::handle(info) -> println!("[panic] ...") -> wait_forever()

Test Mode Panic

Test mode adds one extra responsibility: it must tell the host that the run failed, otherwise the harness cannot distinguish panic from a hang.

panic::handle(info) -> println!("[panic] ...") -> qemu::exit_failure()

Dependency Boundaries

Allowed Dependencies

  • main.rs may call the test runner only when the test-kernel feature is enabled.
  • test/mod.rs may call qemu::exit_success() because it is part of the test harness layer.
  • panic.rs may call qemu::exit_failure() only in test mode.

Disallowed Dependencies

  • General kernel logic should not call qemu::exit_*.
  • Future modules like memory, task, syscall, or filesystem should not know whether they are running under QEMU.
  • qemu::exit_* is not a generic error-handling API. It is only a host-observation mechanism for test infrastructure.
