Allocator Strategy: Stack vs Bitmap

This diagram shows how the physical memory manager keeps the same public API while changing the allocator strategy underneath. The stack allocator stores one address per free frame; the bitmap allocator stores one bit per frame and scans for a free bit.

Stable API

memory::alloc_frame() and memory::dealloc_frame() stay the same for callers.

Strategy Boundary

FrameAllocator defines the common contract for stack and bitmap implementations.

Selected Strategy

pmm.rs currently selects BitmapFrameAllocator with a type alias.

Stack Allocator

Stores every free frame address in a static array. Allocation pops the top address; free pushes it back.

metadata = free frame addresses alloc = pop() free = push(frame)

Bitmap Allocator

Stores one bit per frame. A clear bit means free; a set bit means used.

PMM Dispatch

The strategy is selected inside pmm.rs, while callers continue to use the same functions.

type SelectedFrameAllocator = BitmapFrameAllocator; static mut PMM: SelectedFrameAllocator = SelectedFrameAllocator::new();
main.rs -> memory::alloc_frame() -> PMM.alloc_frame() -> bitmap

Trade-Offs

  • Stack metadata costs one address per free frame, so it is simple but larger.
  • Bitmap metadata costs one bit per frame, so it is compact but allocation may scan.
  • The current bitmap implementation manages the first non-empty segment after kernel reservation.
  • Future multi-region memory maps can extend the bitmap strategy without changing the public PMM API.

Smoke Test Signal

The normal boot path prints the selected allocator and current free count, so allocator behavior is visible from QEMU.
