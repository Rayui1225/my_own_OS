Milestone 4

SBI Timer Interrupt Flow

This diagram explains how the current kernel gains a time concept in supervisor mode. The key idea is simple: ask OpenSBI to schedule the next timer event, catch the resulting supervisor timer interrupt, increment a tick counter, then schedule the next one again.

Privilege Level

Our kernel runs in S-mode, so it does not directly own the machine timer hardware. It asks OpenSBI for timer service.

Main Goal

This milestone is not about scheduling yet. It is about establishing a repeating tick source that later code can build on.

Most Important Pattern

Every timer interrupt ends by scheduling the next timer interrupt. That re-arm step is what keeps the clock alive.

1

Boot Connects Trap and Timer

trap::init(); println!("[debug] trap ready"); println!("[debug] enabling timer"); timer::init();

The boot path first ensures that traps already have a valid entry point. Only then do we turn on timer-related state.

Why this order matters: if the timer interrupt arrived before stvec was configured, the kernel would have nowhere safe to land.
2

Enable Supervisor Interrupt Delivery

csr::enable_supervisor_timer_interrupt(); csr::enable_supervisor_interrupts();

The first call enables the timer interrupt source itself. The second call opens the supervisor interrupt gate globally.

sie.STIE means “allow supervisor timer interrupts.” sstatus.SIE means “interrupts may enter supervisor mode at all.”
3

Pick a Deadline and Ask OpenSBI to Arm It

let deadline = csr::read_time().wrapping_add(TICK_INTERVAL); sbi::set_timer(deadline);

The kernel reads the current hardware time counter, adds a fixed interval, and asks OpenSBI to trigger the next timer interrupt when that deadline is reached.

Key mental model: we do not say “interrupt every second forever.” We say “interrupt me at this next absolute time.”
4

Timer Interrupt Enters the Trap Pipeline

TrapCause::SupervisorTimerInterrupt => { timer::handle_interrupt(); }

Once the deadline is reached, the normal trap machinery runs. The trap layer only identifies the interrupt cause and delegates the actual timer behavior.

Boundary rule: trap does dispatch, timer owns timer semantics.
5

Increment Tick and Re-Arm the Next One

let tick = TICKS.fetch_add(1, Ordering::Relaxed) + 1; println!("[timer] tick = {}", tick); schedule_next_tick();

This is the actual heartbeat of the system. One interrupt advances the global tick count exactly once, prints a visible log, and then schedules the next interrupt.

Without re-arm: you would only ever get one timer interrupt.

Control Flow Summary

kernel_main → timer::init() → read_time() + interval → sbi::set_timer() → SupervisorTimerInterrupt → timer::handle_interrupt()

This is a repeating loop, not a one-shot initialization. The same pattern keeps the system clock moving forward.

Why sepc Is Not Advanced

Timer interrupts are asynchronous

Unlike ebreak or an illegal instruction, the timer interrupt was not caused by “the current instruction being invalid.” It simply interrupted normal execution.

  • Breakpoint: often needs sepc += 4 so we do not re-run the trap-causing instruction.
  • Timer interrupt: usually returns to the same sepc, because that instruction itself was fine.

Why SBI First

S-mode fit

The kernel already runs under OpenSBI in supervisor mode, so using the SBI timer path matches the execution environment we already have.

Focus on concepts

This milestone is about time, interrupts, and re-arm behavior, not about diving into machine-mode timer hardware details yet.

Cleaner layering

The timer module can talk to a small firmware API instead of mixing scheduler prep with platform-specific timer register trivia.

What to Watch For

  • If you enable interrupts but forget the first set_timer(), nothing will tick.
  • If you handle one timer interrupt but forget to schedule the next one, the clock stops after a single event.
  • If you mix timer logic into the trap module, later scheduler work gets harder to reason about.
  • If you treat timer interrupts like synchronous faults and modify sepc incorrectly, control flow will become wrong in subtle ways.
